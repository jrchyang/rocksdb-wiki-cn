## 简介

数据库中的所有数据都按逻辑顺序排列。应用程序可以指定键的比较方法，该方法指定键的总排序序。迭代器 API 允许应用程序对数据库进行范围扫描。迭代器可以查找指定的键，然后应用程序可以一次一个键的从该点开始扫描。迭代器 API 还可用于对数据库中的键进行反向迭代。创建迭代器时，会创建数据库的一致时间点视图。因此，通过迭代器返回的所有键都来自数据库的一致视图。

## 一致性视图

如果给定了 `ReadOptions.snapshot` ，迭代器将返回该快照的数据。如果为 nullptr，迭代器将从创建迭代器时的隐式快照中读取数据。隐式快照通过 [[Iterator#迭代器固定的资源和迭代器刷新|固定资源]] 来保留。无法将此隐式快照转换为显式快照。

## 错误处理

`Iterator::status()` 返回迭代的错误。错误类型包括： I/O 错误、校验和不匹配、不支持的操作、内部错误或其他错误。

如果没有错误，状态为 `Status::OK()` 。如果状态不为 OK，迭代器也将失效。换言之，如果 `Iterator::Valid()` 为真，`status()` 保证为 `OK()` ，因此无需检查 `status()` 即可安全地继续其他操作：

```C++
for (it->Seek("hello"); it->Valid(); it->Next()) {
  // Do something with it->key() and it->value().
}
if (!it->status().ok()) {
  // Handle error. it->status().ToString() contains error message.
}
```

另一方面，如果 `Iterator::Valid()` 为 `false`，则有两种可能性：(1) 数据已经遍历完成。在这种情况下，`status()` 为 `OK()`；(2) 出现错误。在这种情况下，`status()` 不是 `OK()`。如果迭代器无效，检查 `status()` 始终是一种很好的做法。

`Seek()` 和 `SeekForPrev()` 会丢弃之前的状态。

请注意，在 5.13.x 或更早版本中（2018 年 5 月 17 日合并的 https://github.com/facebook/rocksdb/pull/3810 之前），`status()` 和 `Valid()` 的行为曾经有所不同：

- 即使 `status()` 不正常，`Valid()` 也可能返回 `true`，这有时可用于跳过损坏的数据，现已不再受支持。处理损坏数据的预期方式是 `RepairDB()`（参见 `db.h`）
- `Seek()` 和 `SeekForPrev()` 并不总是丢弃先前的状态。`Next()` 和 `Prev()` 并不总是保留非 ok 状态

## 迭代上限和下限

用户可以通过为传递给 `NewIterator()` 的读取选项参数设置 `ReadOptions.iterate_upper_bound` 来指定范围查询的上限。通过设置此选项，RocksDB 不必在上限之后查找下一个键。在某些情况下，可以避免一些 I/O 或计算。在某些特定工作负载下，改进可能非常显著。请注意，它适用于前向和后向迭代。当您使用高于上限的查找键执行 `SeekForPrev()` 时，或者使用高于迭代器上限的最后一个键调用 `SeekToLast()` 时，行为尚未定义，但 RocksDB 不会崩溃。

同样的，`ReadOptions.iterate_lower_bound` 可以与后向迭代一起使用，帮助 RocksDB 优化性能。

请参阅选项的注释以了解更多信息。

## 迭代器固定的资源和迭代器刷新

迭代器本身不会占用太多内存，但它会阻止某些资源被释放。这包括：

1. 自迭代器创建时起的 memtables 和 SST 文件。即使某些 memtables 和 SST 文件在刷新或压实后被删除，如果迭代器固定了它们，它们仍会保留下来
2. 当前迭代位置的数据块。这些块将保存在内存中，要么固定在块缓存中，要么在未设置块缓存的情况下保存在堆中。请注意，虽然块通常很小，但在某些极端情况下，如果值的大小非常大，单个块可能会非常大

所以迭代器的最佳用途是保持其短暂的生命周期，以便及时释放这些资源。

迭代器有一定的创建成本。在某些用例（尤其是仅内存用例）中，人们希望通过重用迭代器来避免其创建成本。在使用迭代器时，请注意，如果迭代器过期，就会阻碍资源的释放。因此，如果迭代器在一段时间（例如一秒钟）后未被使用，请确保销毁或刷新迭代器。在 5.7 版之前，当需要处理过时的迭代器时，需要销毁迭代器并在必要时重新创建。从 5.7 版开始，可以调用 API `Iterator::Refresh()` 来刷新迭代器。调用该函数后，迭代器将被刷新以代表最近的状态，而之前固定的陈旧资源将被释放。

## 前缀迭代

前缀迭代器允许用户在迭代器中使用布隆过滤器或哈希索引以提高性能。但是该功能有局限性，如果使用不当，可能会返回错误的结果而不报错，因此我们建议您谨慎使用此功能，请参阅 [[Prefix-Seek|前缀查找]] 查看如何使用该功能。选项 `total_order_seek` 和 `prefix_same_as_start` 仅适用于前缀迭代。

## 预读

RocksDB 在迭代过程中如果注意到同一个表文件有超过 2 次 IO，就会自动预读并预取数据，这仅适用于基于块的表格式。预读大小从 8KB 开始，并随着每次额外的连续 IO 而呈指数增加，最大可达 `BlockBasedTableOptions.max_auto_readahead_size`（默认 256 KB）。这有助于减少完成范围扫描所需的 IO 数量。仅当 `ReadOptions.readahead_size = 0`（默认值）时，才会启用此自动预读。在 Linux 上，缓冲 IO 模式下使用 readahead syscall，直接 IO 模式下使用 AlignedBuffer 来存储预取数据。(从 5.12 开始，缓冲 IO 和 5.15 直接 IO 可使用自动迭代器读取前置数据）。

如果你的整个用例都以迭代为主，并且依赖操作系统的页面缓存（即使用缓冲 IO），你可以选择通过 `DBOptions.advise_random_on_open = false` 来打开页面缓存的内部预读。如果在硬盘或远程存储上运行，这将更有帮助，但对直接连接的固态硬盘设备可能没有什么实际影响。

`ReadOptions.readahead_size` 在 RocksDB 中为非常有限的用例提供预读支持。此功能的限制是，如果打开，迭代器的常量成本将高得多，所以你应该只在迭代非常大范围的数据并且无法使用其他方法解决它时使用。典型的用例是存储是具有非常长延迟的远程存储，操作系统页面缓存不可用并且将扫描大量数据。通过启用此功能，每次读取 SST 文件都会根据此设置预读数据。请注意，一个迭代器可以打开每个级别的每个文件，以及同时打开所有 L0 文件，您需要为它们计算好预读内存，并且无法自动跟监控读缓冲区使用的内存。

如果设置了 `ReadOptions.async_io` 标志，RocksDB 可以异步读取数据，包括自动读取和使用 `ReadOptions.readahead_size` 固定读取。在这种情况下，迭代器会尝试在不阻塞迭代线程的情况下预取数据。详情请参阅 [[Asynchronous-IO|异步 IO]] 。

我们正在寻求改进 RocksDB 的预读功能。
## 多列族迭代器（实验性）

MultiCfIterator （CoalescingIterator 和 AttributeGroupIterator）允许用户遍历多个列族的键。它支持大多数现有的迭代器功能。有关如何使用该功能，请参阅 [[Multi-Column-Family-Iterator|多列族迭代器]] 。
