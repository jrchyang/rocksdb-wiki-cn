RocksDB 常用于大规模分布式存储系统。在这些系统中，许多独立的 RocksDB 实例被存储在不同数据中心的成千上万台主机中。在下列情况下，会出现 RocksDB 代码/配置/数据兼容性问题：

- 将 RocksDB 代码升级到新版本，然后重新打开数据库
- 将 RocksDB 代码降级到低版本，然后重新打开数据库
- 将一个数据库的数据复制到另一台主机上，这台主机可能运行不同版本的 RocksDB 和/或 设置不同的 RocksDB 配置

我们并不认为这只是 RocksDB 用户的问题，我们在开发 RocksDB 的新功能时，总是会考虑到潜在的兼容性问题。

## 格式兼容性（Format Compatibility）

下面是我们处理格式兼容性的目标：如果用户不更改数据库的选项配置，在打开由不同版本生成的数据库时，RocksDB 的兼容性保证如下：

- 向后兼容：新版本的 RocksDB 应能打开所有先前版本生成的数据库，进行正常配置
- 向前兼容：如果用户没有明确启用旧版本不支持的新功能，那么旧版本的 RocksDB 应该能够打开至少高几个小版本的版本所生成的 DB。我们的目标是尽可能长久地保持这种向前兼容性，通常可以保持一年以上

例如，如果不使用新功能，5.6 版可以打开至少由 2.2 版（4 年前）生成的数据库，而 3.10 版（2 年前）可以打开使用 5.6 版生成的数据库。我们有持续的测试来验证这一点。

## 选项验证（Option Verification）

当用户从未对 RocksDB 的配置进行破坏性改动时，格式兼容性就能解决问题。然而，如果使用不同的配置打开数据库，即使使用相同的代码，RocksDB 也可能无法打开。想想比较器或压实策略的改变。

为了解决这个问题，RocksDB 可以在 DB 目录下的 [[Options-File|RocksDB 选项文件]] 中将当前的 DB 选项具体化。在打开数据库之前，用户可以选择从数据库中读取选项，然后使用该选项打开数据库，这样就可以兼容了。或者，用户可以验证他们想要使用的新选项是否与当前数据库兼容。

这一功能存在已知的局限性，我们正在加以解决。一个主要问题是，上一节提到的前后兼容性保证不适用于选项验证工具。例如，如果新版本的 RocksDB 有了划分元数据块的新功能，那么选项文件中就会出现 "meta block size = ... "的新记录。即使用户使用默认值打开该功能，旧版本的 RocksDB 也无法理解这条记录，用户只能认为它不兼容。因此，降级后的 RocksDB 版本可能无法使用此功能。

为了解决这个问题，我们在 5.6 版中为选项验证工具添加了一个新参数，以跳过未识别的选项。有了这项功能，用户就可以打开使用更高版本生成的选项文件了。我们正在努力解决其他限制。

## 迁移工具（Migration Tool）

检测选项不兼容性是一大进步。然而，用户有时需要更多。他们并不总是满足于 RocksDB 告诉他们 "不"。他们希望 RocksDB 能够让其变成可以 ---- 如果他们愿意支付成本，就会自动迁移数据。

我们创建了一个压实设置迁移工具，帮助用户将 DB 迁移到不同的压实设置。请参见 [tool definition](https://github.com/facebook/rocksdb/blob/main/include/rocksdb/utilities/option_change_migration.h)。如果用户有需要，我们可能会改进该工具，使其包含更多应用场景。欢迎您为此发送 PR 请求，或告诉我们您的要求。
